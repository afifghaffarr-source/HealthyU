# Plan: Decouple HealthyU from Lovable AI Platform

> Generated: 2026-06-13 | Mode: Architect / Plan
> Status: Approved by user (4 decisions confirmed)

## Goal

Remove every "Lovable lock-in" in the codebase. Project should run on a
self-managed Supabase project, deploy to Cloudflare Pages + Workers, use
VexoAPI for AI, and log errors to a custom Supabase table.

## Decisions (confirmed)

1. **Deploy target**: Cloudflare Pages + Workers
2. **Supabase**: export data from Lovable Cloud → import into new standalone project
3. **AI provider**: VexoAPI (gpt-oss-120b default, glm-4.7-flash for cheap, gemini via VexoAPI for multimodal)
4. **Error reporting**: custom logger to Supabase `error_reports` table

## Phases

### Phase 1 — Auth rewrite (Lovable → Supabase direct)
**Why first**: auth is a hard dependency for testing the rest; can't run end-to-end without it.

**Changes**:
- `package.json`: remove `@lovable.dev/cloud-auth-js`
- `src/integrations/lovable/index.ts`: DELETE file (use `supabase.auth.signInWithOAuth` directly)
- `src/routes/auth.tsx`: replace `lovable.auth.signInWithOAuth("google", ...)` → `supabase.auth.signInWithOAuth("google", ...)`
- `client.ts` / `client.server.ts` / `auth-middleware.ts` / `auth-attacher.ts`: keep file, change header comment from "auto-generated by Lovable" → "self-managed Supabase client". Code unchanged.

**Files touched**: 3 (delete 1, modify 2). Net diff: ~15 lines.

**Risk**: low. Supabase OAuth with same provider is a drop-in. Test plan: bunx tsc --noEmit, manual test in dev mode.

### Phase 2 — Error reporting rewrite
**Why second**: same reasoning as auth — small, isolated, removes a hard dependency.

**Changes**:
- `src/lib/lovable-error-reporting.ts`: DELETE
- New file: `src/lib/errorReporting.ts` — calls `supabase.from("error_reports").insert(...)` (fire-and-forget, like the original Lovable version)
- `src/components/healthyu/global-error-boundary.tsx`, `feature-error-boundary.tsx`, `route-boundaries.tsx`: update import
- New migration: `supabase/migrations/20260613000000_error_reports.sql` — minimal `error_reports` table with RLS (service-role write, no public read)

**Files touched**: 5 (1 new, 1 migration, 1 delete, 3 imports updated). Net diff: ~50 lines.

**Risk**: low. Old behavior was a no-op when `window.__lovableEvents` was undefined (Lovable dev). New behavior writes to Supabase. We add a `NODE_ENV === "development"` short-circuit to avoid filling the table during dev.

### Phase 3 — AI gateway rewrite (Gemini → VexoAPI)
**Why third**: this is the biggest behavioral change, deserves a focused PR after auth/error are done.

**Changes**:
- `src/features/ai/lib/aiGateway.server.ts`: replace `fetch(GEMINI_AI, ...)` block with VexoAPI adapter. Public API (`callAiWithGuards`, `callAiJsonWithGuards`, `callAiJsonWithSchema`, `AiGatewayError`, `extractJsonFromResponse`) unchanged → all 100+ callsites keep working.
- `src/features/ai/lib/aiStreamGateway.server.ts`: same treatment for streaming path.
- `src/features/ai/lib/aiBudget.server.ts`: model price table → VexoAPI prices. We don't have per-model rates, so set a conservative default ($0.001/1k input, $0.003/1k output for gpt-oss-120b, lower for glm-4.7-flash).
- `src/features/ai/lib/aiRouter.server.ts`: model name mapping. "google/gemini-2.5-flash" → "gpt-oss-120b", "google/gemini-2.5-pro" → "gemini" (VexoAPI's own multimodal gemini endpoint). Emergency tier stays on gemini.
- `src/features/moderation/lib/imageModeration.server.ts`, `src/features/content/lib/contentGeneration.server.ts`: update any direct model references.
- `.env.example`: rename `GEMINI_API_KEY` → `VEXO_API_KEY`, add `VEXO_BASE_URL=https://vexoapi.dev`.

**Files touched**: ~6. Net diff: ~200 lines (mostly in gateway, plus model name map).

**Risk**: medium. VexoAPI response shape differs (`{status, data, timestamp}` instead of OpenAI's `{choices: [...]}`). The adapter needs careful conversion. JSON mode may not be supported the same way — VexoAPI returns text in `data`, we prompt the model with "respond in JSON" and parse.

**Fallback strategy**: if VexoAPI fails (e.g. upstream 403 like we hit today), gateway returns 503 with clear error. We can add a `AI_FALLBACK_URL` env later to swap to OpenAI/Anthropic without code change.

### Phase 4 — Vite config migration
**Why fourth**: required for Cloudflare Pages to actually build the app.

**Changes**:
- `package.json`: remove `@lovable.dev/vite-tanstack-config` dev dep, add raw plugins: `@tanstack/react-start`, `@vitejs/plugin-react`, `nitropack` (for cloudflare preset), `vite-tsconfig-paths`
- `vite.config.ts`: rewrite to use `defineConfig` from `vite` directly with the plugins needed for TanStack Start + Cloudflare target
- `package.json` scripts: `build` → `vite build` (unchanged); the framework plugin handles SSR/Worker output

**Files touched**: 2. Net diff: ~30 lines.

**Risk**: medium. `@lovable.dev/vite-tanstack-config` is opinionated — it bundles tailwind, tsconfig paths, error logger, dev sandbox detection, and the tanstack/router component tagger. We'll need to re-add these manually or live without the dev-only ones (componentTagger is dev-only and not required).

**Caveat**: Cloudflare Workers target requires Nitro `cloudflare-pages` preset. The current config likely uses a `cloudflare` default target. Verify by reading the source or trying build.

## Risk Summary

| Risk | Mitigation |
| ---- | ---------- |
| VexoAPI key returns 403 (current state) | Document key rotation; design `AI_FALLBACK_URL` swap for future |
| `@lovable.dev/vite-tanstack-config` has hidden behavior | Build local config, test `bun run build` end-to-end before merge |
| VexoAPI doesn't have real streaming | Buffer full response, return as single chunk — UX degradation noted |
| Image multimodal limited in VexoAPI gpt-oss-120b | Use VexoAPI's "gemini" endpoint for image tasks, fall back to text-only with explicit user prompt to "describe what's in the image" |
| Supabase export from Lovable Cloud may miss RLS / triggers | Use Lovable's "Backup" or pg_dump — both covered in `docs/supabase-migration.md` |
