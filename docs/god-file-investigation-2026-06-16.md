## Investigation 2: `src/components/ui/sidebar.tsx` (745 LOC) — DEAD CODE

**Diagnostic signals (fatal red flag):**

- **0 importers** anywhere in `src/`
- 0 tests reference any export
- File is **100% stock shadcn/ui template** (verbatim from `pnpm dlx shadcn@latest add sidebar`)
- Real app uses `src/components/healthyu/desktop-sidebar.tsx` (38 LOC, custom, the only one imported)

**Action taken:** `git rm`. 745 LOC deleted. Commit `a5d1e738`.

**Why splitting would be anti-pattern:**

- shadcn convention is single-file for the entire `SidebarProvider` system
- `useSidebar` hook is shared by 14+ subcomponents (tight coupling)
- Splitting breaks "copy-paste shadcn update" workflow
- If team ever re-installs via `shadcn add sidebar`, file comes back automatically

## Investigation 3: `src/integrations/supabase/types.ts` (6474 LOC) — AUTO-GENERATED

**Diagnostic signals (fatal red flag):**

- Header: `__InternalSupabase: { PostgrestVersion: "14.5" }` — this is Supabase CLI output
- Content: ~50 database tables with `Row`, `Insert`, `Update`, `Relationships` types
- 10 importers, all using single `Database` type alias for `createClient<Database>` call
- Standard Supabase TypeScript pattern (https://supabase.com/docs/guides/api/generating-types)

**Action taken:** Marked as **N/A** in roadmap. **Do not refactor.**

**Why splitting would break things:**

- `supabase gen types typescript` regenerates the file as a unit — splitting would break the regeneration workflow
- The `Database` type alias is the public API; all `createClient<Database>` calls depend on it
- Auto-generated types follow schema-relationship inference (you can't safely split a normalized database schema by file boundary)
- No benefit: TS types are erased at build time, so "smaller files" doesn't improve bundle/runtime

## Lesson Learned (saved to skill `god-file-domain-split`)

**Always verify "god-file" assumption FIRST** with these 3 checks:

1. **Grep for importers** — `grep -rln "<file-name>" src/`. If 0 importers, it's **dead code** (delete, not refactor).
2. **Check for "auto-generated" markers** — headers like `// Generated by X`, `__InternalSupabase:`, `// DO NOT EDIT`. If present, it's **generated code** (untouchable, will be regenerated).
3. **Look for re-export patterns** — many `X.functions.ts` files re-exporting from one `Y.functions.ts` means Y is a **consolidation target** (real god-file, split candidate).

**Cost of skipping this check (anti-pattern refactor):**

- Splitting shadcn template → breaks `useSidebar` context coupling, breaks update workflow
- Splitting auto-generated code → breaks regeneration, creates manual maintenance burden

**Time spent on this investigation: ~30 min, all 3 verdicts actionable.**
